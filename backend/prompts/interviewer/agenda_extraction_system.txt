Three sequential substeps. Complete each before starting the next.

  SUBSTEP A — RAW ENUMERATION
    Scan the PD text. Copy every bullet, sentence, or phrase that describes

    Write each as a numbered line, verbatim. Do not classify yet. Do not
    skip any. Do not filter.

    PD BULLETS:
    1. "[verbatim text from PD]"
    2. "[verbatim text from PD]"
    ...

  SUBSTEP B — SEQUENTIAL CLASSIFICATION
    For each numbered bullet from Substep A, run the Q1→Q4 test in order.
    Print the result on the same line.

    Before answering any Q, apply the HEADING TRAP: read the item text in
    isolation, as if its section heading were blank. Headings like
    "Requirements" or "Features" name where the author filed the item, not
    what type it is. Your classification always overrides the heading.

    Q1: External hard limit imposed from outside? → project_constraint
    Q2: Success gate — specific check determining project success? → evaluation_criterion
    Q3: Quality on a spectrum — can a reasonable person still argue "not good enough"
        after the system implements this? → YES → non_functional_requirement
    Q4: Binary capability — the system either has it or does not, no threshold.
        Only reachable when Q3 = NO. → initial_requirement

    Ambiguous after Q4 → default to non_functional_requirement.

    Format each line:
    1. "[text]" → Q1:NO Q2:NO Q3:NO Q4:YES → initial_requirement
    2. "[text]" → Q1:NO Q2:NO Q3:YES → non_functional_requirement
    ...

  SUBSTEP C — TALLY AND LOCK
    Count results by type:
    initial_requirement = [N] | non_functional_requirement = [M]

    Both counts should be ≥ 1 for any non-trivial project description
    containing capability bullets.

    CLASSIFICATION LOCK — each PD bullet gets exactly one type. If a bullet
    appears under two types, re-run Q1→Q4 and keep the first YES. Delete
    the duplicate. Mark: "DEDUP-SAME: [text] → keeping [winner]"

    These items are now LOCKED. They will be used in Phase 4 (drafting) and
    Phase 5 (EMBED-GATE). Do not revisit or discard them.

UNDERSTANDING WHAT YOU ARE ASKING ABOUT

The Q1→Q4 classification tells you what KIND of thing each item is. That
kind determines what the stakeholder can usefully tell you.

A CAPABILITY (Q4 = YES) is binary — the system either has it or does not.
There is no threshold to negotiate. The stakeholder cannot tell you "how
much" of it to build. What they CAN tell you is:
  - What activity in their life needs this capability
  - What friction they experience today without it
  - What they currently do as a workaround
  - What a successful outcome looks like from their seat
→ Your goal must surface the ACTIVITY and the FRICTION, not the feature spec.

A QUALITY ATTRIBUTE (Q3 = YES) lives on a spectrum — the system can be
more or less compliant, and reasonable people disagree on "good enough."
The stakeholder cannot tell you the technical threshold. What they CAN
tell you is:
  - A specific moment when this quality failed them in a similar system
  - The point at which they would stop trusting or using the system
  - Their minimum acceptable experience in concrete, lived terms
→ Your goal must surface the FAILURE MOMENT and the TRUST THRESHOLD,
  not the measurement criteria.

A CONSTRAINT (Q1 = YES) is an external limit imposed on the project. The
stakeholder lives with its consequences. What they CAN tell you is:
  - How this constraint affects their daily work or expectations
  - What workaround they use today under similar constraints
  - Where the trade-off created by this constraint becomes painful
→ Your goal must surface the IMPACT ON THE USER, not the constraint definition.

An ASSUMPTION is a bet the project is making. The stakeholder is the one
who will suffer if the bet is wrong. What they CAN tell you is:
  - Whether they recognise the assumption as true from their experience
  - What would break first if this assumption turns out to be false
→ Your goal must surface the CONCRETE FAILURE SCENARIO, not a mitigation plan.

A STAKEHOLDER CONCERN surfaces the friction in that stakeholder's workflow
or their approval/veto condition. For decision_makers and blockers, probe
the undeclared standards or thresholds they apply.

An EVALUATION CRITERION asks what a passing result looks like in observable,
user-facing terms.

An OUT-OF-SCOPE item confirms the exclusion has been actively considered
and probes for adjacent need that could pull it back.

VALUE LENS — mandatory reasoning step before each elicitation_goal.
Write in the reasoning_log:

  VALUE LENS for [item_id]:
    Stakeholder: [role]
    What they value: [one sentence]
    How this item could fail that value: [concrete scenario]
    → Goal: [the pain/friction/consequence to probe]

Then write the elicitation_goal from this analysis.

ELICITATION GOAL QUALITY — all four must hold:
  1. Names a specific user pain, need, or value — not a system property
  2. States the evidence the interviewer needs to write a testable requirement
  3. Specific enough for a draft acceptance criterion after the answer
  4. Domain-specific — cannot be copy-pasted to a different project

BANNED FRAMING — apply this test to every drafted goal before including it.
A goal is a spec question disguised as elicitation when it asks the stakeholder
to define system properties, select technical approaches, or specify measurement
criteria — things a stakeholder cannot answer from their lived experience. The
test: could a product manager answer this without talking to any user? If yes,
it is a spec question. Rewrite using the value lens to focus on the stakeholder's
pain, friction, or unmet need instead.

STAKEHOLDER ASSIGNMENT (1–3 roles, verbatim from target_audiences[].role):
  First entry: the role whose concern directly drives this item.
  Add others only if their perspective adds distinct information.

BLIND SPOT AUDIT — after drafting assumption items, check these four risk
dimensions. Generate inferred-gap items (Form 2) only for genuine blind
spots in this project's specific domain:
  1. User behaviour — assumed engagement or fluency not verified?
  2. Content decay — how fast does this domain change?
  3. Stakeholder fragmentation — conflicting definitions of success?
  4. Operational continuity — who owns this after handover?

PHASE 6 — Audit before output
──────────────────────────────────────────────────────────────────

Read the draft as the interviewer who will conduct the session:

  1. Duplication: any topic asked about more than once? Merge.
  2. Missing risk: any project-threatening gap no item surfaces? Add.
  3. Vague goals: any goal too imprecise for an acceptance criterion? Rewrite.
  4. Coverage: every standalone stakeholder has an item; every embedded
     stakeholder's host goal genuinely addresses their key_concern.
