──────────────────────────────────────────────────────────────────
Step 1 — Primary Pain and Underlying Tension
──────────────────────────────────────────────────────────────────

Who experiences the strongest pain? What is happening today that fails them?

Identify the TWO FORCES pulling in opposite directions that create this
pain. Every real problem contains a tension — two things that conflict and
cannot both be fully satisfied. Name that tension explicitly.

If you cannot name a tension, your problem statement is too shallow. A
problem solvable by "building a website" alone is not sharp enough.

Then for each stakeholder you have identified so far, write one sentence
describing what they VALUE most in the context of this project. This is not
a need or a feature — it is the underlying outcome they care about. This
value map will inform all downstream elicitation.

──────────────────────────────────────────────────────────────────
Step 2 — Success Condition
──────────────────────────────────────────────────────────────────

What must change for the solution to succeed? Define a concrete end-state
in user terms — what the user can do after that they could not do before.

──────────────────────────────────────────────────────────────────
Step 3 — Boundaries
──────────────────────────────────────────────────────────────────

What is explicitly required, constrained, or excluded? List these before
writing project_constraints and out_of_scope to avoid omissions.

──────────────────────────────────────────────────────────────────
Step 4 — Capability and Quality Scan
──────────────────────────────────────────────────────────────────

Scan every bullet, sentence, or phrase in the project description that
describes what the system should do or how it should behave.

For each, determine whether it is:
  [CAPABILITY] — a discrete system behaviour: the system either has it or does not.
  [QUALITY]    — a spectrum attribute: reasonable people can disagree on "good enough".

Write each as a tagged line in the reasoning_log:
  [CAPABILITY] "<quoted or paraphrased capability from the PD text>"
  [QUALITY]    "<quoted or paraphrased quality attribute from the PD text>"

These items do NOT belong in the Vision schema. They are extracted by the
downstream agenda agent directly from the project description. Note them
here only so you do not accidentally absorb them into Vision fields.

──────────────────────────────────────────────────────────────────
Step 5 — Stakeholder Inventory (Phase A — read-only)
──────────────────────────────────────────────────────────────────

Using SEMANTIC READING, scan the full project description and enumerate
every individual stakeholder named or described.

INDIVIDUAL vs. CATEGORY — mandatory distinction:
  Category labels ("Primary users:", "Secondary users:") are taxonomy
  headers. They determine the TYPE field but are NOT entries themselves.
  Each named individual or group WITHIN a category is one entry.

  If the PD groups multiple named roles under a category label, each named
  role is one entry. The category label determines the TYPE field but is
  NOT an entry itself.

  For example, if the PD lists three named roles under a "Secondary users:"
  heading → three entries, all typed secondary_user.
  If the PD lists one named group under "Primary users:" → one entry,
  typed primary_user.

Two rules, no exceptions:
  - Do NOT merge two individually named stakeholders into one entry.
  - Do NOT add a stakeholder not named in the project description.

Write as a numbered list. Record the count:

  STAKEHOLDER INVENTORY
  1. ...
  2. ...
  STAKEHOLDER TARGET: [N]

This number is a hard constraint. target_audiences must contain exactly N.

1. STAKEHOLDERS — Phase B (Enrichment)

Begins after Phase A is complete and STAKEHOLDER TARGET is recorded.

For each stakeholder in the inventory, assign:
  - role: exact name from the project description. Do not rename.
  - type: one of [primary_user, secondary_user, beneficiary, decision_maker, blocker]

    If the PD groups stakeholders under labeled categories, treat the PD's
    grouping as strong evidence for type. Record the mapping:
      "PD groups [role] under '[label]' → type=[selected_type]"

    If no grouping exists, apply in order (stop at first match):
      primary_user    — main audience, uses the system regularly and directly.
      secondary_user  — uses the system occasionally or in a supporting role.
                        Performs actions on the system but is not the primary audience.
      beneficiary     — benefits from outputs without directly using the system.
      decision_maker  — approves scope, budget, standards, or direction.
      blocker         — can veto or block the project.

  - influence_level: [high, medium, low]
  - key_concern: the single most important friction or need for this role,
    grounded in the project description.

DIFFERENTIATION: no two stakeholders may share the same key_concern.
  primary_user    concerns → what they can DO (actions, capabilities)
  secondary_user  concerns → tasks they perform THROUGH the system
  beneficiary     concerns → outcomes they receive without direct use
  decision_maker  concerns → consistency, liability, institutional risk
  blocker         concerns → conditions that must be met before approval

If differentiation is insufficient from the PD, write the best available
concern and note: "Insufficient information to differentiate [role] —
requires validation." Never remove a stakeholder for lack of information.

CROSS-CHECK (mandatory, in reasoning_log):
  STAKEHOLDER CROSS-CHECK:
  Phase A target: [N]
  Phase B written: [M]
  Status: MATCH / MISMATCH

If MISMATCH: identify, correct, and record what changed before proceeding.

2. CORE PROBLEM

1-2 sentences describing the core problem.
  - Name the UNDERLYING TENSION (two competing forces), not just a symptom.
  - Specific enough to reject features that do not address the tension.
  - Do not describe the solution or reference the system being built.

---

3. VALUE PROPOSITION

The outcome delivered to users when the core problem is resolved.
  - What changes FOR THE USER, not what the system does.
  - Must directly address the tension in core_problem.
  - No feature lists. No technology names. No marketing language.

---

4. CONSTRAINTS, EVALUATION CRITERIA, AND SCOPE

YOUR SCOPE BOUNDARY — what belongs in Vision vs. what goes downstream:

Vision captures: core problem, value proposition, stakeholders, assumptions,
constraints, evaluation criteria, and scope boundaries.

Feature capabilities and quality attributes are NOT Vision fields. They are
extracted downstream by the agenda agent from the PD text directly. You
already tagged them in Step 4. Do not place them in any Vision field.

FIELD PLACEMENT — for every candidate item, apply this decision tree:

  First ask: "Is this a Project Boundary (constraint, exclusion) or a
  Definition of Success (evaluation criterion, assumption)?"
  → YES → place in the appropriate Vision field. Stop.
  → NO  → check the two delegation tests:

    Test A: "Can this be implemented and declared done — binary capability?"
    → YES → downstream (capability). Do not place in Vision.

    Test B: "Quality on a spectrum — can someone argue 'not good enough'?"
    → YES → downstream (quality attribute). Do not place in Vision.

  If an item matches neither Vision nor delegation → note in reasoning_log.

FIELD RULES:
  evaluation_criteria  — specific checks or tests declared in the PD.
                         Extract verbatim. Do NOT invent criteria.
  project_constraints  — hard delivery conditions imposed externally.
                         Do not limit to specific patterns.
  A single item goes into exactly ONE field. If it matches both,
  project_constraints wins (harder external limit takes precedence).

---

5. ASSUMPTIONS

Implicit beliefs that must be true for the solution to work but are NOT
confirmed in the project description.

For each candidate:
  Step 1: "What are we betting on that we have not verified?"
  Step 2: "What is the most damaging way this could be FALSE?"
          If minor or easily fixed → discard.
  Step 3: "Is this already confirmed in the PD?" If yes → constraint, not assumption.

Generate at least one per category:

  A — User Engagement:
    Does the audience have motivation and opportunity to use this?
  B — Content Validity Over Time:
    Will the information stay accurate given how fast the domain changes?
  C — Stakeholder Alignment:
    Do all decision-makers share the same definition of success?
  D — Operational Continuity:
    Who owns this after handover, and do they have capacity to maintain it?

For each assumption:
  statement:         Clear declarative sentence. No vague qualifiers.
  risk_if_wrong:     One sentence naming the specific impact.
  needs_validation:  true if a false assumption would materially change
                     scope, architecture, or user trust.

---

6. OUT OF SCOPE

Capabilities excluded from the system.
  - Definitive language: "This system will not..."
  - Focus on exclusions a stakeholder might expect to be included.
  - Minimum 2 items. Do not pad with trivia.
